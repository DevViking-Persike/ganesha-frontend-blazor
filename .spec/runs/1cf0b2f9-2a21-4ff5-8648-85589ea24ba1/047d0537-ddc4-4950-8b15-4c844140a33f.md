# Diagnóstico Arquitetural — Frontend Ganesha DesignLab

**Papel:** `Analyze` · **Etapa:** Diagnóstico Arquitetural · **Data:** 2026-07-05 · **Base:** código atual em `src/` + `tests/`

> ⚠️ **Nota de atualização:** Os três docs de partida do grafo (`01-mapeamento-inicial.md`, `03-design-system.md`, `05-diagnostico-testes.md`) e o `02-diagnostico-arquitetural.md` prévio **estão defasados**. Esta entrega corrige o diagnóstico para refletir o código atual. Os pontos de drift estão listados em §8.

## 1. Estilo arquitetural (verificado)

Arquitetura em **camadas leves host-thin + shared library forte**, sobre **Blazor Server (.NET 10)** com renderização interativa server-side. O host web é apenas bootstrap/pipeline/roteamento; praticamente toda a massa de UI vive na Razor Class Library `Ganesha.DesignLab.Shared`. Estilo predominante: **biblioteca de Design System + catálogo de padrões (lab)** — não é aplicação de produto.

Evidência:
- `src/Ganesha.DesignLab.Web/Program.cs:1-40` — host com 41 linhas, só DI + pipeline HTTP + mapping.
- `src/Ganesha.DesignLab.Web/Program.cs:38` — `AddAdditionalAssemblies(typeof(...Shared._Imports).Assembly)` conecta a shared no roteamento.
- `src/Ganesha.DesignLab.Shared/DependencyInjection/ServiceCollectionExtensions.cs:9-17` — composition root `AddGaneshaDesignLab()`.

## 2. Mapa de responsabilidades por camada (estado atual)

| Camada | Responsabilidade | Localização |
|---|---|---|

## 3. Fluxos de dados representativos (validados)

**Fluxo A — alternância de tema:** `LabLayout` → `IThemeService.SetTheme()` → evento `OnThemeChanged` → `GnsThemeProvider` atualiza `data-theme` no `<html>` → CSS semântico `[data-theme="dark"]` troca os tokens. Evidência: `Services/ThemeService.cs`, `Components/Infrastructure/GnsThemeProvider.razor`.

**Fluxo B — toast:** `LabFeedback` → `IToastService.ShowSuccess(...)` → `ToastService` adiciona `ToastMessage` (lock + `List`), emite `OnToastAdded`, agenda `Timer` de auto-dismiss → `GnsToastContainer` renderiza e remove. Evidência: `Services/ToastService.cs:6-66`, `Components/DesignSystem/Feedback/GnsToastContainer.razor`.

**Fluxo C — chart interativo:** componente Razor renderiza SVG estático + `data-*` → JS global (`ganesha-charts.js`, gerado por esbuild a partir de `ts/charts/*.ts`) anexa tooltips/observadores via `MutationObserver`. Evidência: `ts/charts/tooltip.ts`, `ts/core/observer.ts`, `DesignSystem/Charts/GnsBarChart.razor`.

## 4. Problemas arquiteturais encontrados

### P1 — Violação de Regra 1 (tamanho de arquivo) — **Alta severidade**
- `Components/Lab/Pages/LabCharts.razor`: **941 linhas** (teto ~500).
- `Components/Lab/Pages/LabGrid.razor`: **510 linhas** (acima do teto).
- `Components/Composites/ProjectTimeline/GnsProjectTimeline.razor`: **376 linhas** (zona de atenção, já é composite — candidato a split).
**Impacto:** revisão difícil, acoplamento alto, fusão de showcase+mock+layout. `LabCharts.razor` mistura ~50 estilos inline + 200+ linhas de dados mockados (`_monthlyRevenue`, `_topProducts`, `_weeklyVisitorsSeries`, linhas 724-941).

### P2 — Showcase acoplado a dado mockado inline — **Média-alta**
`LabDashboard.razor`, `LabTablePage.razor` e `LabCharts.razor` declaram `record`s privados (`OrderRecord`, `UserRecord`) e coleções mockadas **dentro do `@code`** do `.razor`. Não existe separação em `*Samples.cs`/`*MockData.cs`.
**Impacto:** (a) baixa testabilidade de padrões; (b) ambiguidade entre "contrato de UI" e "exemplo de dados"; (c) quando um feature real chegar, o esforço de separar estado de view será pago de uma vez.

### P3 — Camada de integração/domínio/dados ausente — **Média (esperada para o estágio)**
Confirmado: **zero** `HttpClient`, `IHttpClientFactory`, `DbContext`, adapters, repositories ou models de negócio. `Models/` contém só tipos de UI.
**Impacto:** a arquitetura foi validada apenas para UI estática + estado local; quando features reais chegarem, será preciso introduzir `application/`+`domain/`+`infrastructure/` (Regra 4). **Não é dívida hoje** — é a postura correta para um design lab (Regra 5: não antecipar abstração).

### P4 — Fronteira DS × tela por convenção, não por barreira — **Média**
Composites novos (`GnsProjectTimeline`, `GnsInteractiveChart`) coexistem com páginas-lab que replicam composições ricas inline. Não há régua automatizada (grep/teste) que impeça lógica de feature de entrar em `Lab/Pages`.
**Impacto:** risco de "showcase vira dono de contrato", contrariando `.claude/rules/frontend-arquitetura.md`.

### P5 — Host MAUI divergente — **Baixa-média (já conhecido)**
`Maui/Components/Pages/` ainda é template default (`Counter.razor`, `Weather.razor`, `Home.razor`, `NotFound.razor`), sem consumir o DS `Gns*`. Mantém-se o achado do diagnóstico prévio (FE-006 do improvement-plan persiste).
**Impacto:** narrativa arquitetural dupla; custo cognitivo; decisões de DS não refletem no MAUI.

### P6 — Subecossistema técnico de charts sem governança formal — **Média**
13 componentes de chart (Bar, Bubble, Donut, Funnel, Gauge, HorizontalBar, Line, Pie, Process, StackedBar, ProgressBar, RadialProgress, Sparkline) dependem de SVG + JS global via interop, com build TS/esbuild separado. É a única parte do DS com toolchain não-.NET.
**Impacto:** superfície de regressão maior; regras de a11y (`role="img"`, `<title>`) precisam de validação específica; o build `npm run build` precisa ser gates do CI para evitar `ganesha-charts.js` stale.

### P7 — Dívida técnica pontual em serviço de UI — **Baixa**
`Services/ToastService.cs:55-58` cria `new Timer(...)` e descarta com `_ = timer` (timer não é rastreado nem disposado por toast). Funciona, mas vaza timers até o GC em cenários de alta rotatividade.
**Impacto:** pequeno hoje; relevante apenas quando o lab virar produto.
